# WORKSPACE-FINAL-FIX — единый интерфейс рабочего пространства для всех поездок

## Почему возникала подмена интерфейса

В `features/integration-controller.js` существовала функция `appRenderGenericTripPanels(appState)`.
Она сохраняла оригинальный полноценный интерфейс вкладок только для поездки `trip-turkey-2026`,
а для всех остальных tripId заменяла содержимое `panel-overview`, `panel-route`, `panel-docs`,
`panel-members`, `panel-monitor`, `panel-messages` и `panel-settings` через большие строки `innerHTML`
на упрощённые статические generic-панели. Из-за этого Минск, Казань, новые и завершённые поездки
теряли карточки, модальные окна, обработчики и мобильную адаптацию.

## Как удалена generic-реализация

- `features/integration-controller.js` переписан: функции `appRenderGenericTripPanels()` больше нет,
  как нет и любых условий вида `if (trip.id === "trip-turkey-2026") { ... } else { ... }`.
- `appRenderAll()` для любого tripId выполняет один и тот же набор шагов: шапка, аватары,
  панель «Обзор», панель «Маршрут», сегменты SOS, глобальные кнопки, статус поездки.
- Панели документов, участников, мониторинга, сообщений и настроек вообще не пересоздаются
  контроллером: их обновляют собственные модули через подписку на общее состояние.
- Проверено: `grep -rn "appRenderGenericTripPanels"` по production-коду — 0 совпадений;
  `trip-turkey-2026` в контроллере не упоминается ни разу.

## Как оригинальные вкладки сделаны общими

- Все поездки используют исходную DOM-разметку `trip-overview.html`. Панели никогда не заменяются
  целиком через `innerHTML` при изменении состояния: контроллер точечно обновляет текст и списки
  (через сравнение прежнего HTML — `appSetHtmlIfChanged`), а модули documents / members / monitoring /
  messages / settings рендерят свои области собственными идемпотентными render-функциями.
- Заголовок поездки, маршрут, даты, подзаголовок «Участники поездки»,селектор сегментов в модальном окне SOS,
  countdown до ближайшего события и роль вкладки «Документы» стали data-driven (обновляются из состояния
  активной поездки, без данных Турции).
- Пустые поездки показывают честные empty states внутри тех же карточек: «Сегменты маршрута пока
  не добавлены», «Документы пока не добавлены», «В поездке пока нет других участников», «Активных
  сигналов и изменений пока нет», пустое состояние списка сообщений.

## Как данные связаны с tripId

- Источник данных — каталог `trips` / `completedTrips` в `assets/js/app-state-bridge.js`: у каждой
  поездки свои `segments`, `workspaceDocuments`, `participants`/`participantIds`/`roles`, `logistics`,
  `weather`, `routePreview`, `nextEvent`.
- Добавлены нормализованные срезы по tripId: `coreFlowByTripId` (сигналы, нарушения, Plan B, сообщения,
  история мониторинга), `invitationsByTripId`, `offlineCopyByTripId`. Они входят в PERSIST_KEYS
  и сохраняются между сессиями.
- При `setActiveTrip(tripId)` мост собирает рабочий срез (`workspacePatchForTrip`): документы,
  участники, приглашения, offline-копия и coreFlow строго выбранной поездки. Обратные записи модулей
  (`documents`, `participants`, `invitations`, `coreFlow`, `offlineCopy`) перехватываются и сохраняются
  в срез именно активной поездки — смешивание данных между поездками исключено.
- Роль вычисляется по сочетанию currentUserId + tripId (`trip.roles[userId]`), а не глобально:
  Артём — организатор Турции и Казани, но участник Минска.
- Для каждой поездки формируются ровно три варианта Plan B с корректным источником данных
  («Источник данных поездки «…»» / «Архив поездки «…»» для завершённых).

## Какие поездки проверены (headless Chromium, все 7 вкладок каждой)

1. `trip-turkey-2026` — эталон, активная, организатор.
2. `trip-minsk-2026` — активная, текущий пользователь участник (управляющие действия скрыты).
3. `trip-kazan-2026` — активная соло-поездка, организатор.
4. Новая поездка, созданная через мастер-поток (`createTrip`) — честные empty states.
5. `done-tbilisi` — завершённая: собственный архив (2 сегмента, 3 документа, 4 участника,
   подтверждённое нарушение, опубликованный Plan B, сообщения и история) в режиме read-only.

Дополнительно проверены: переключение поездок без перезагрузки страницы, режим No Access
(DOM поездки, название и маршрут не раскрываются), офлайн-режим (SOS disabled, модальные окна
изменений заблокированы), «Завершить поездку» (переход в Историю, повторная активация запрещена),
«Удалить поездку» (подтверждение вводом УДАЛИТЬ, экран удаления и переход на Главную),
«Удалить офлайн-копию» (поездка сохраняется).

## Какие файлы изменены

- `features/integration-controller.js` — переписан без generic-панелей и ветвлений по trip.id.
- `assets/js/app-state-bridge.js` — данные поездок (сегменты, документы, логистика, погода Турции и
  Казани, архив Тбилиси), срезы `coreFlowByTripId` / `invitationsByTripId` / `offlineCopyByTripId`,
  перехват записей модулей по активной поездке, удаление поездки с очисткой срезов.
- `assets/js/core-flow-state-adapter.js` — убраны все хардкоды демо-данных Турции из рабочих
  сценариев (черновик Plan B, история, подписи), экспорт демо-набора для эталонной поездки.
- `trip-overview.html` — data-driven countdown, синхронизация роли вкладки «Документы», честный
  empty state документов, id для подзаголовка участников, нейтральный подзаголовок OCR-окна.
- `features/trip-members.js` — синхронизация режима организатор/участник при смене поездки.
- `features/trip-settings.js` — реальный переход на Главную после удаления поездки (AppRoutes).
- `docs/FILE-MAP.md` — обновлено описание integration-controller.
- Новое: `docs/WORKSPACE-FINAL-FIX.md`, `screenshots/workspace-final/` (15 скриншотов).

## Результаты браузерных тестов

- Console errors: 0; page errors: 0; unhandled promise rejections: 0 (во всех прогонах).
- Duplicate ID: не обнаружено ни на одной поездке.
- Горизонтальный overflow: 0 px на desktop 1440×900 и mobile 390×844.
- Все 7 вкладок видимы и наполнены для всех 5 проверенных поездок; данные Турции не «протекают»
  в другие поездки (сканирование текстовых узлов DOM).
- Модальные окна и обработчики работают; в offline/read-only режимах модальные окна изменений
  не открываются, SOS заблокирован.

## Оставшиеся ограничения

- Это frontend-демо без backend: данные живут в localStorage/sessionStorage; «источники данных»
  мониторинга — демонстрационные.
- OCR, импорт из почты и Telegram-уведомления — имитация потоков без реальных интеграций.
- Погода задана данными поездки (для новых поездок блок скрывается/показывает прочерки),
  реального API погоды нет.
- Демо-переключатели состояний внутри вкладок (например, сценарии документов) сохранены как
  инструмент демонстрации и не влияют на изоляцию данных.
