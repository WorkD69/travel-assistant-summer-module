# Security policy

## Сообщение об уязвимости

Не публикуйте рабочие токены, персональные данные и детали эксплуатации в issue.
Передайте владельцу проекта минимальное воспроизведение приватным каналом и
укажите компонент, влияние и безопасный способ проверки.

## Секреты

- `.env`, cookies, ключи, SQLite-базы, логи и platform metadata не коммитятся.
- Используйте разные случайные `JWT_SECRET` и `BOT_SERVICE_TOKEN` для каждого
  окружения.
- AI-ключ хранится только в variables целевой платформы.
- `TELEGRAM_BOT_TOKEN` должен находиться только в окружении единственного
  polling-процесса бота и не переноситься в backend.
- После случайной публикации секрет считается скомпрометированным и должен быть
  немедленно отозван, даже если файл удалён последующим коммитом.

## Минимальные меры эксплуатации

- Явный HTTPS origin для CORS, без `*`.
- TLS на внешних URL.
- Резервное копирование SQLite до изменения схемы.
- Ограничение доступа к Volume и VPS.
- Один polling-процесс на Telegram token.
- Периодическая ротация токенов и проверка Gitleaks.

## Границы

Это учебный демонстрационный проект, не прошедший независимый pentest. Он не
должен хранить документы или персональные данные реальных путешественников без
отдельной модели угроз, политики хранения, шифрования, антивирусной проверки
загрузок и юридической оценки.
