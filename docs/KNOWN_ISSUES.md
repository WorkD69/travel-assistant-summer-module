# Известные ограничения

- Plan B использует Mock GDS. Реальные бронирования, билеты, места, возвраты и
  платежи не изменяются.
- Telegram consumer подключён к B2 staging backend, но на момент фиксации
  `WEB_APP_BASE_URL` действующего VPS ведёт на старый production frontend.
  Кнопка deep link может открыть не B2 preview. В этой публикации окружение бота
  намеренно не менялось.
- SQLite подходит для демонстрации и одного instance, но ограничен по
  параллельной записи и горизонтальному масштабированию.
- Загруженные документы хранятся как SQLite blobs. Нет production-grade
  antivirus/DLP pipeline, object storage lifecycle и полной политики удаления.
- OCR зависит от качества скана, языка, раскладки и PDF; результат требует
  проверки пользователем.
- Open-Meteo/geocoding зависят от внешней доступности, лимитов и точности данных.
- AI зависит от Groq-compatible endpoint, модели, квот и качества prompt.
  Ответы могут ошибаться и не являются юридической или медицинской консультацией.
- Offline/PWA cache может потребовать ручного refresh после публикации; не все
  write actions доступны offline.
- Приглашения одноразовые и имеют срок действия, но email delivery не реализован.
- Password recovery представлен интерфейсом, без полноценной production email
  delivery.
- Нет интеграции с live carrier/GDS status; monitoring частично основан на
  демонстрационных или пользовательских данных.
- Mobile QA покрывает основные экраны, но не все устройства, браузеры и
  accessibility сценарии.
- Политики retention/удаления аккаунта, централизованный audit monitoring,
  alerting и независимый security review требуют отдельной работы.
- Для одного Telegram token допустим только один polling-процесс; второй вызовет
  `409 Conflict`.
